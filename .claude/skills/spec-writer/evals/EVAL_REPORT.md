# spec-writer 評価レポート

`skill-creator` の評価ループ（draft → test → grade → improve → retest）で
`spec-writer` スキルを評価・改良した記録。

## 方法

- **対象**: `spec-writer`（機能要望 → 実装可能な技術設計仕様）
- **実行モデル**: Opus（セッションと同一）。ユーザが実際に体験する条件で測定。
- **比較**: `with_skill`（スキルを読んで作業）vs `without_skill`（スキル無しの素のモデル）。
  両者ともリポジトリ（GeoSection 実コード）にアクセス可能 → 差分はスキルの指針のみ。
- **テストケース 3 件**（`evals.json`）:
  1. `kml-import` — GPX に加え KML 取り込み（GeoSection ドメイン）
  2. `multiday-merge` — 複数日 GPX を 1 本の連続断面図に（GeoSection ドメイン）
  3. `expense-ocr-bot` — 経費 Slack ボットにレシート OCR 下書き（**リポジトリ外** = 過適合検出用）
- **各設定 2 runs**（分散把握）。
- **2 つの指標**:
  - **アサーション合格率**（定量）: 独立した採点エージェントが各仕様を期待項目に照合。
  - **盲検 A/B**（定性）: どちらの設定で書かれたか伏せた審査員が、より実装可能な方を選ぶ。

## イテレーション 1（v1）

| 指標 | with_skill (v1) | baseline |
|---|---|---|
| アサーション合格率 | 100% | 97% |
| 盲検 A/B 勝利数 | **1 / 3** | **2 / 3** |

**所見**: アサーションは 18 個中 16 個が両設定とも合格 → ほぼ非判別（強い Opus baseline は
構造的に完成した仕様を既に書ける）。意味のある信号は盲検 A/B にあり、v1 は **2-1 で敗北**。
審査員の理由づけが 2 つの根本原因を特定:

1. **ヘッジ（決め切らない）**: 主要な技術判断を「未解決の論点」へ先送り
   （しきい値「0.6 程度」「TBD」）。baseline は `0.80`・リトライ回数・目標精度 85%・
   p95 ≤25s と具体値で確定。
2. **再利用を正しさより優先**: multiday で既存の距離関数を再利用（"N=1 で綺麗"）した結果、
   日境界で距離が飛び傾斜が滲む副作用を見落とし。baseline は日ごと再ベース化という
   正しい難しい方を選択。

なお v1 が**勝った** kml-import では、スキルの核（実コードへの接地・無言バグの優先・
回帰カバレッジ）が評価された → 接地は強み、commitment と decision-quality が穴。

## v1 → v2 の変更

- 新原則 **「決め切る。人間/プロダクト判断が要る点だけ open に隔離する」**（具体値必須）。
- 新原則 **「load-bearing な決定を特定し、再利用の都合でなく正しさから決める」**
  （multiday の境界バグを実例として明記）。
- 接地（実コード参照）の強みは維持。
- 評価の判別性向上のため、各ケースに「確定 vs 先送り」を測る判別性の高い期待項目を 1 つ追加。

## イテレーション 2（v2）

| 指標 | iter-1 (v1) | iter-2 (v2) |
|---|---|---|
| 盲検 A/B: with_skill 勝利 | 1 / 3 | **2 / 3** |
| multiday-merge（v1 敗北） | baseline 9.0 | **with_skill 9.0 vs 8.0** ✅ 逆転 |
| expense-ocr-bot（v1 敗北） | baseline 9.0 | **with_skill 9.0 vs 7.5** ✅ 逆転 |
| kml-import | with_skill 8.7 | baseline 8.9（−0.5、両者とも秀逸） |
| アサーション合格率 | 100% / 97% | 98% / 98% |

**結論**: v2 は盲検 A/B を **1-2 → 2-1 に反転**。v1 が負けた 2 ケースを、こちらが狙った軸
（具体値の確定・境界判断の正しさ）で審査員に評価され、いずれも明確な差で勝ち取った。
残る 1 敗（kml-import）は審査員が両者「秀逸」とした上での 0.5 差で、ノイズ域。

## 注意・限界

- **アサーション指標は飽和**（両設定 ~98%）。強モデル + 実コードアクセスでは構造的完成度に
  差が出にくい。本スキルのような主観的成果物では盲検 A/B が主指標、アサーションは下限保証。
- **盲検 A/B は各ケース n=1**（イテレーションあたり 3 比較）。数値は小標本だが、審査理由が
  v1→v2 で一貫した因果（ヘッジ解消・境界判断の改善）を示しており、確度は高い。
- **時間/トークンは未計測**（Workflow ハーネスでは per-run 計測が取れず、benchmark の
  Time/Tokens 列は 0）。比較は品質軸（合格率・勝率）に限定。
- 1 件のグレーダ出力が placeholder で返ったため該当 run を再採点（7/7）して補正済み。

## 今後の選択肢

- **より大規模な確認**: テストケースを増やす / 各ケースの比較を複数審査員・複数 run 化して
  盲検 A/B の確度を上げる（skill-creator の "expand the test set" 段階）。
- **kml-import の詰め**: baseline はコード差分の明示と追加エッジ（非有限トークンの除去・
  連続アップロードの stale state）で僅差勝ち。スキルに「変更関数のコード差分を示す」「逐次
  操作の状態ハザードを点検する」を軽く足す案（過適合に注意）。
- **トリガー最適化**: `skill-creator` の `run_loop.py`（`claude -p` 利用）で description の
  発火精度を最適化（本体が固まってから実施）。
